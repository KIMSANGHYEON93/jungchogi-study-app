## PART 1: 디자인 패턴 개요

### 1-1. 디자인 패턴이란?

- 소프트웨어 설계에서 **반복적으로 발생하는 문제에 대한 재사용 가능한 해결책**
- GoF(Gang of Four)가 23개 패턴으로 분류

### 1-2. 3가지 분류

| 분류 | 개수 | 목적 | 암기법 |
|------|------|------|--------|
| **생성** (Creational) | 5개 | 객체 **생성** 방식 | **빌프팩앱싱** |
| **구조** (Structural) | 7개 | 클래스/객체 **조합** | **어데퍼브플컴프** |
| **행위** (Behavioral) | 11개 | 객체 간 **상호작용** | **스템메옵스커이미비체인인** |

## PART 2: 생성 패턴 (5개) — "빌프팩앱싱"

### Builder (빌더)
> 복잡한 객체를 **단계별로** 생성

- 생성 과정과 표현을 분리
- 동일 공정으로 다른 결과물 생성

### Prototype (프로토타입)
> 기존 객체를 **복제(clone)**하여 새 객체 생성

- 원형(prototype)을 복사 후 수정
- `clone()` 메서드 사용

### Factory Method (팩토리 메서드) ★★★★
> 상위 클래스에서 인터페이스 정의, **하위 클래스에서 인스턴스 생성**

- 객체 생성을 서브클래스에 위임
- "어떤 클래스를 만들지는 자식이 결정"

### Abstract Factory (추상 팩토리) ★★★
> 관련된 **객체군(family)**을 생성하는 인터페이스

- 팩토리의 팩토리
- 서로 연관된 객체들을 묶어서 생성

### Singleton (싱글톤) ★★★★★ (최빈출!)
> 클래스의 인스턴스를 **오직 하나만** 보장

- `private` 생성자 + `static` 인스턴스
- Day 2 문제 5에서 이미 코드 등장!

```java
// 시험에 나오는 싱글톤 코드 패턴
class Singleton {
    private static Singleton instance = null;
    private Singleton() {}   // private 생성자!
    public static Singleton getInstance() {
        if (instance == null)
            instance = new Singleton();
        return instance;
    }
}
```

## PART 3: 구조 패턴 (7개) — "어데퍼브플컴프"

### Adapter (어댑터) ★★★★
> 호환되지 않는 인터페이스를 **변환**하여 함께 사용

- 110V → 220V 변환기처럼 중간 매개 역할
- 기존 코드 수정 없이 재사용 가능

### Decorator (데코레이터) ★★★★
> 객체에 **동적으로 기능을 추가** (소스 변경 없이)

- 상속 대신 조합(composition) 사용
- 기능을 덧씌워서 확장

### Facade (퍼사드) ★★★★
> 복잡한 시스템에 **단순한 인터페이스** 제공

- 건물의 정면(facade)처럼 하나의 창구
- 느슨한 결합 제공

### Bridge (브리지) ★★★
> **추상부와 구현부를 분리**하여 독립적 확장

- 추상 계층과 구현 계층을 분리
- 결합도를 낮춤

### Flyweight (플라이웨이트) ★★
> 대량의 작은 객체를 **공유**하여 메모리 절감

- 공통 부분을 클래스화하여 공유
- 메모리 효율 극대화

### Composite (컴포지트) ★★★
> **트리 구조**로 부분-전체 계층 표현

- 개별 객체와 복합 객체를 동일하게 취급
- 파일/폴더 구조가 대표적 예시

### Proxy (프록시) ★★★
> **대리인**이 실제 객체에 대한 접근을 제어

- 실제 객체 대신 대리 객체가 로직 수행
- 접근 제어, 로깅, 캐싱 등에 활용

## PART 4: 행위 패턴 (11개)

### Strategy (전략) ★★★★
> 다양한 알고리즘을 **캡슐화**하여 교체 가능

- 동일한 문제를 다른 알고리즘으로 해결
- 런타임에 알고리즘 변경 가능

### Template Method (템플릿 메서드) ★★★★
> **알고리즘의 골격(구조)을 정의**, 세부 단계는 서브클래스에서 구현

- 상위 클래스에서 전체 흐름 제어
- 변하지 않는 부분은 상위, 변하는 부분은 하위

### Mediator (중재자) ★★
> 객체 간 **상호작용을 중앙화**

- 객체들이 직접 통신하지 않고 중재자를 통해 소통
- 채팅방의 서버 역할

### Memento (메멘토) ★★
> 객체의 **상태를 저장**해두었다가 **복구** (Undo)

- 이전 상태로 되돌리기 기능
- Ctrl+Z 기능의 패턴

### Observer (옵저버) ★★★★★ (최빈출!)
> 객체 상태 변화 시 **의존 객체들에게 자동 통지**

- 일대다(1:N) 의존 관계
- "구독-알림" 모델 (유튜브 구독 알림)
- **발행-구독(Publish-Subscribe)** 패턴이라고도 함

### State (상태) ★★★
> 객체 **내부 상태에 따라 행위가 변경**

- 상태를 객체로 표현
- if-else 대신 상태 객체 위임

### Visitor (방문자) ★★
> 객체 구조 변경 없이 **새로운 연산(기능) 추가**

- 데이터 구조와 연산을 분리

### Command (커맨드) ★★★
> **요청 자체를 객체로 캡슐화**

- 실행될 기능을 객체화
- 매개변수로 전달, 큐에 저장 가능

### Iterator (반복자) ★★★
> 컬렉션 내부 구현을 노출하지 않고 **순차적으로 접근**

- for-each 패턴의 근간
- `hasNext()`, `next()` 메서드

### Chain of Responsibility (책임 연쇄) ★★
> 요청을 처리할 수 있는 객체가 **연결 고리로 전달**

- 한 객체가 처리 못하면 다음으로 넘김

### Interpreter (인터프리터) ★
> **언어의 문법 규칙**을 클래스로 표현

- SQL, 정규표현식 등의 해석기

```

  어(Adapter) 데(Decorator) 퍼(Facade) 브(Bridge) 플(Flyweight) 컴(Composite) 프(Proxy)

  스(Strategy) 템(Template Method) 메(Mediator) 옵(Observer) 스(State)
  커(Command) 이(Iterator) 미(Memento) 비(Visitor) 체인(Chain of Responsibility) 인(Interpreter)
```

## PART 6: UML (Unified Modeling Language)

### 6-1. UML 다이어그램 분류

| 분류 | 다이어그램 | 설명 |
|------|-----------|------|
| **구조** (정적) | 클래스 | 클래스 속성/메서드/관계 ★★★★★ |
| | 객체 | 특정 시점의 객체 인스턴스 |
| | 컴포넌트 | 소프트웨어 구성 요소 간 관계 |
| | 배치 (Deployment) | 물리적 HW에 소프트웨어 배치 |
| | 패키지 | 패키지 간 관계 |
| | 복합체 구조 | 클래스 내부 구조 |
| **행위** (동적) | 유스케이스 | 사용자 관점 기능 표현 ★★★★★ |
| | 시퀀스 | 객체 간 메시지 흐름 (시간순) ★★★★ |
| | 커뮤니케이션 | 객체 간 메시지 + 관계 |
| | 활동 (Activity) | 업무 흐름 (순서도와 유사) |
| | 상태 (State) | 객체 상태 변화 |
| | 타이밍 | 시간 제약 표현 |

> **시험 포인트**: "구조적 다이어그램 6개, 행위적 다이어그램 6개"로 구분!

### 6-2. UML 관계 (6가지) ★★★★★

| 관계 | 기호 | 설명 | 예시 |
|------|------|------|------|
| **연관** (Association) | ─── | 클래스 간 개념적 연결 | 학생 ── 과목 |
| **집합** (Aggregation) | ◇── | 부분이 **독립** 존재 가능 | 팀 ◇── 선수 |
| **합성** (Composition) | ◆── | 부분이 전체에 **종속** | 건물 ◆── 방 |
| **의존** (Dependency) | - - -> | 일시적 사용 관계 | 주문 - - -> 결제 |
| **일반화** (Generalization) | ──▷ | **상속** 관계 | 동물 ──▷ 고양이 |
| **실체화** (Realization) | - -▷ | **인터페이스 구현** | Flyable - -▷ Bird |

> **집합 vs 합성 차이**:
> - **집합(◇)**: 팀이 해산해도 선수는 존재 → 독립적
> - **합성(◆)**: 건물이 무너지면 방도 사라짐 → 종속적

### 6-3. 유스케이스 다이어그램 구성 요소

| 요소 | 설명 |
|------|------|
| **액터 (Actor)** | 시스템 외부 사용자/시스템 (사람 아이콘) |
| **유스케이스** | 시스템이 제공하는 기능 (타원) |
| **관계** | 아래 표 참조 |

**유스케이스 관계:**

| 관계 | 키워드 | 설명 |
|------|--------|------|
| **포함** | `<<include>>` | **반드시** 실행되는 기능 |
| **확장** | `<<extend>>` | **선택적**으로 실행되는 기능 |
| **일반화** | | 상위-하위 유스케이스 |

> **시험 함정**: include = **필수**, extend = **선택**!

### 6-4. 시퀀스 다이어그램 구성 요소

| 요소 | 설명 |
|------|------|
| **생명선 (Lifeline)** | 객체의 존재 기간 (점선) |
| **활성 박스** | 객체가 활동 중인 구간 |
| **메시지** | 객체 간 상호작용 (화살표) |
| **동기 메시지** | ──▶ (실선 꽉 찬 화살표) |
| **비동기 메시지** | ──> (실선 열린 화살표) |
| **반환 메시지** | - - -> (점선) |

### 6-5. 클래스 다이어그램 접근 제어자

| 기호 | 접근 제어자 | Java 키워드 |
|------|------------|-------------|
| `+` | public | `public` |
| `-` | private | `private` |
| `#` | protected | `protected` |
| `~` | package | `(default)` |

## PART 7: 결합도 & 응집도 (매회 출제!)

### 7-1. 결합도 (Coupling) — 낮을수록 좋음!

| 결합도 | 설명 | 수준 |
|--------|------|------|
| **자료 결합도** | 데이터(매개변수)만 전달 | **최선** ↑ |
| **스탬프 결합도** | 배열, 객체 등 자료구조 전달 | |
| **제어 결합도** | 제어 신호(flag) 전달 | |
| **외부 결합도** | 외부 데이터 공유 | |
| **공통 결합도** | 전역 변수 공유 | |
| **내용 결합도** | 다른 모듈 내부 직접 접근 | **최악** ↓ |

> **암기**: **자스제외공내** (자→스→제→외→공→내, 좋은것→나쁜것)

### 7-2. 응집도 (Cohesion) — 높을수록 좋음!

| 응집도 | 설명 | 수준 |
|--------|------|------|
| **기능적 응집도** | 하나의 기능만 수행 | **최선** ↑ |
| **순차적 응집도** | 출력이 다음 입력으로 사용 | |
| **통신적 응집도** | 같은 데이터 사용 | |
| **절차적 응집도** | 순서대로 실행 | |
| **시간적 응집도** | 같은 시간에 실행 | |
| **논리적 응집도** | 유사 기능을 모아놓음 | |
| **우연적 응집도** | 관련 없는 기능이 모임 | **최악** ↓ |

> **암기**: **우논시절통순기** (우→논→시→절→통→순→기, 나쁜것→좋은것)
